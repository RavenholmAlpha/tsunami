# Surge：分层拥塞控制

> [English](surge.md) | **中文** | [日本語](surge.ja.md)

## 概述

Surge 是 TSUNAMI 的自适应连接扩展系统。它根据实时流并发量管理底层 TLS 连接数量，在单连接效率和多连接吞吐量之间取得平衡。

### 为什么需要 Surge？

单连接多路复用对隐蔽性来说是理想的（一个连接 = 一个 HTTPS 会话）。但在高负载（多个并发流）下，单个 TLS 连接会成为瓶颈：

- **队头阻塞**：一个流上的慢速 TLS 写入会阻塞所有其他流
- **TCP 公平性**：一个连接只获得一份"公平份额"的带宽；多个连接可以聚合带宽
- **保活开销**：一个连接上的多个空闲流会产生不必要的帧处理

Surge 通过根据实际需求在单连接（Layer 1）和多连接（Layer 2）之间动态切换来解决这个问题。

## 设计原则

1. **默认单连接** — 在低负载时最小化资源使用和指纹
2. **自动扩展** — 仅在需要时升级到多连接
3. **无数据包重排** — 每个流在其整个生命周期内固定在一个 TLS 连接上
4. **单端口** — 所有连接使用相同端口（通常为 443）
5. **渐进关闭** — 空闲连接仅在可配置的空闲超时后关闭

## 分层架构

### Layer 1（默认）

所有流在单个 TLS 连接上多路复用。

```
Stream A ─┐
Stream B ─┤──► Session 0 ──► TLS Connection 0 ──► :443
Stream C ─┘
```

**激活条件**：并发流数 ≤ 阈值（默认：8）

### Layer 2（自动升级）

当并发流数超过阈值时，打开新的 TLS 连接，使用最小负载策略在连接之间分配流。

```
Stream A ─┐
Stream B ─┤──► Session 0 ──► TLS Connection 0 ──┐
Stream C ─┘                                      │
                                                  ├──► :443
Stream D ─┐                                      │
Stream E ─┤──► Session 1 ──► TLS Connection 1 ──┘
Stream F ─┘
```

**激活条件**：并发流数 > 阈值

### 层级转换

```
         流数 > 阈值
Layer 1 ─────────────────────────► Layer 2
         ◄─────────────────────────
         流数 ≤ 阈值/4
         且会话数减少到 1
```

- **升级**（1 → 2）：当活跃流数超过阈值时触发。控制器主动异步创建额外会话，最多 `max-connections`。
- **降级**（2 → 1）：当活跃流降至 `threshold/4` 或以下时触发（更宽的滞后以防止快速振荡）。空闲会话（空闲时间超过 `idle-downgrade-time`）被关闭，至少保留 1 个。仅当会话数达到 1 时才恢复 Layer 1。

控制器每 **2 秒**评估一次层级转换。

### 升级行为

升级到 Layer 2 时，Surge 不仅仅创建一个额外连接 — 它**主动扩展**到 `max-connections`：

```
evaluate() 检测到流数 (13) > 阈值 (8)
  → 存储 Layer 2
  → 异步：创建会话直到池中有 max-connections (4)
    → Session 2 已创建 (2/4)
    → Session 3 已创建 (3/4)
    → Session 4 已创建 (4/4)
```

如果在 Layer 2 期间流压力持续，控制器在每个评估周期继续尝试扩展，直到达到 `max-connections`。

## 配置

| 参数 | 默认值 | 描述 |
|:----------|:--------|:------------|
| `mode` | `auto` | 运行模式：`none`（仅单连接）或 `auto` |
| `threshold` | `8` | 触发 Layer 2 的并发流数 |
| `max-connections` | `4` | Layer 2 中最大 TLS 连接数 |
| `idle-downgrade-time` | `30s` | 会话在降级关闭前必须空闲多长时间 |

### 客户端 CLI

```bash
./tsunami-client \
  --server example.com:443 \
  --password "..." \
  --max-connections 4 \
  --threshold 8
```

### 调优指南

| 场景 | 建议 |
|:---------|:---------------|
| 轻量浏览（< 8 标签页） | 默认设置（threshold=8, max=4）效果良好 |
| 大量并发下载 | 将阈值增加到 16；将 max-connections 增加到 8 |
| 移动端/计量连接 | 将模式设为 `none` 以保持单连接 |
| 高延迟链路 | 将阈值降低到 4 以更快升级 |

## 会话选择

### Layer 1

使用标准会话池逻辑：优先选择空闲会话（最新优先），回退到任何非关闭会话进行多路复用。仅当没有会话存在时才创建新会话。

### Layer 2

使用**最小负载**分配：新流被分配到活跃流最少的会话。如果没有可用会话，则创建新会话（最多 `max-connections`）。

这确保了连接间的均匀流分布，无需主动迁移（流永远不会在会话之间移动）。

## 会话池

会话池（`pkg/mux`）管理所有 TLS 连接的生命周期：

| 行为 | 详情 |
|:---------|:-------|
| **空闲检查** | 每 30 秒，关闭空闲超过 60 秒的会话（至少保留 1 个） |
| **序列跟踪** | 每个会话有单调递增的序列号用于排序 |
| **自动创建** | 当没有合适的会话可用时，池会拨号新的 TLS 连接 |
| **优雅关闭** | 活跃流自然完成；仅空闲会话有资格被清理 |

## 线路协议

Surge 使用 `SURGE_CTRL`（`0x0B`）帧进行控制信令：

| 动作码 | 名称 | 描述 |
|:------------|:-----|:------------|
| `0x01` | `ReportThroughput` | 报告当前吞吐量指标 |
| `0x02` | `RequestMoreConn` | 请求客户端打开额外连接 |
| `0x03` | `ReduceConn` | 信号客户端减少连接数 |
| `0x04` | `BandwidthLimit` | 设置带宽限制 |

带宽统计通过 `BW_REPORT`（`0x0C`）帧定期交换。

## 与替代方案的比较

| 方法 | 优点 | 缺点 |
|:---------|:-----|:-----|
| **始终单连接** | 最小指纹，简单 | 队头阻塞，带宽瓶颈 |
| **始终多连接** | 高吞吐量 | 大指纹，低负载时浪费资源 |
| **Surge（自适应）** | 两者兼得：低负载时隐蔽，高负载时高吞吐 | 层级转换逻辑略有复杂度 |
