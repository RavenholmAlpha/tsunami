# プログラマブルパディングシステム

> [English](padding.md) | [中文](padding.zh.md) | **日本語**

## 概要

TSUNAMI のプログラマブルパディングシステムにより、サーバーはセッションの最初の N パケットのパディングと分割方法を動的に制御できます。パディングスキームは `cmdUpdatePaddingScheme`（`0x06`）を通じてサーバーからすべての接続クライアントにプッシュされ、クライアント側の更新なしにトラフィックパターンを変更できます。

### なぜプログラマブルパディングか？

従来のプロキシプロトコルは固定のパケットサイズとタイミングパターンを持ち、統計分析でフィンガープリントが可能です。TSUNAMI は異なるアプローチを取ります：

- **サーバー制御**：パディングルールはすべてサーバーが定義し、クライアントではありません。調整は 1 RTT 以内にすべての接続クライアントに伝播します。
- **パケットごとの粒度**：最初の N パケットそれぞれが独自のサイズ分布とセグメンテーションルールを持てます。
- **ランタイム更新可能**：サーバーは `UPDATE_PADDING` コマンドを通じていつでも新しいパディングスキームをプッシュできます。クライアントは即座に適用します。
- **アイドル期間カモフラージュ**：バックグラウンドのキープアライブフレームがアイドル期間中もトラフィックの存在を維持し、「沈黙フィンガープリント」を防止します。

## スキームフォーマット

パディングスキームは改行区切りのキーバリューペアを含むテキストブロックとして定義されます：

```
stop=8
0=30-30
1=100-400
2=400-500,c,500-1000,c,500-1000,c,500-1000,c,500-1000
3=9-9,500-1000
4=500-1000
5=500-1000
6=500-1000
7=500-1000
keepalive=30000-60000:4-8
```

## キー

### `stop`

```
stop=<N>
```

最初の N パケット（0 インデックス）の後にパディングルールの適用を停止します。インデックス ≥ N のパケットは変更なしに送信されます。

**デフォルト**：`8`

### パケットインデックスルール

```
<index>=<segment>[,<segment>,...]
```

指定されたインデックスのパケットがどのようにセグメント化されるかを定義します。各パケットインデックスは、送信データの分割とパディング方法を記述するセグメントのリストにマッピングされます。

### セグメントタイプ

| タイプ | 構文 | 説明 |
|:-----|:-------|:------------|
| サイズ範囲 | `min-max` | `[min, max]` バイトのランダムサイズのセグメントを発出 |
| チェック | `c` | すべてのユーザーデータが消費された場合、セグメントの発出を停止 |

**例**：`2=400-500,c,500-1000,c,500-1000`

パケットインデックス 2 の場合：
1. 400–500 ランダムバイトのセグメントを発出（実データ + パディングで充填）
2. **チェック**：ユーザーデータが残っていなければ停止 — パケット完了
3. 500–1000 バイトのセグメントを発出
4. **チェック**：ユーザーデータが残っていなければ停止
5. 最終の 500–1000 バイトのセグメントを発出

チェック（`c`）メカニズムにより、短いペイロードは小さなパケットを生成し、長いペイロードはセグメント化されたフレーム全体を埋めることができます。これにより統計分析に耐える自然な分散が生まれます。

### `keepalive`

```
keepalive=<interval_min>-<interval_max>:<size_min>-<size_max>
```

アイドル期間のバックグラウンドキープアライブパケットを設定します。セッション内のすべてのストリームがアイドル状態のとき、クライアントはランダムな間隔で小さな `cmdWaste` パディングフレームを送信します。

| パラメータ | 説明 |
|:----------|:------------|
| `interval_min`–`interval_max` | キープアライブパケット間のランダム間隔（ミリ秒） |
| `size_min`–`size_max` | ランダムキープアライブパケットサイズ（バイト） |

**例**：`keepalive=30000-60000:4-8` はアイドル時に 30–60 秒ごとに 4–8 バイトのウェイストフレームを送信します。

## デフォルトスキーム

組み込みのデフォルトスキーム：

```
stop=8
0=30-30
1=100-400
2=400-500,c,500-1000,c,500-1000,c,500-1000,c,500-1000
3=9-9,500-1000
4=500-1000
5=500-1000
6=500-1000
7=500-1000
keepalive=30000-60000:4-8
```

このスキームは：
- 各セッションの最初の 8 パケットをパディング
- 最初のパケットが正確に 30 バイトであることを保証（認証パディングサイズに一致）
- パケット 2–7 にチェックポイント付きの可変長セグメンテーションを適用
- アイドル期間中に 30–60 秒ごとに微小なキープアライブフレームを送信

### チューニングガイドライン

| シナリオ | 調整 |
|:---------|:-----------|
| 高帯域幅サーバー | `stop` を増加（例：12–16）、より大きなサイズ範囲を使用 |
| 低帯域幅/モバイル | `stop` を減少（例：4）、オーバーヘッドを減らすためより小さなサイズ範囲を使用 |
| 高セキュリティ環境 | `c` チェックポイント付きの追加セグメント、より短いキープアライブ間隔 |
| CDN フロントデプロイ | パディングサイズを典型的な HTTP/2 レスポンスパターンに合わせる |

## スキームバージョニング

各スキームは生のテキストの **MD5 ハッシュ**で識別されます。クライアントは `SETTINGS` フレームに現在の `padding-md5` を含めます。サーバーのスキームが異なる場合、`cmdUpdatePaddingScheme` を通じて新しいスキームをプッシュします。

これにより再接続時に変更されていないスキームの不要な再送信を回避します。

## ワイヤプロトコル

### 新スキームのプッシュ

サーバーは `cmdUpdatePaddingScheme`（`0x06`）を送信し、フレームデータとして生のスキームテキストを含みます：

```
Frame {
    Command:  0x06 (UPDATE_PADDING)
    StreamID: 0
    Data:     <scheme text bytes>
}
```

クライアントはスキームを解析し、MD5 を計算し、すべての将来のパケットに即座に適用します。

### パディングフレーム

パディングは `cmdWaste`（`0x00`）フレームを使用して実装されます：

```
Frame {
    Command:  0x00 (WASTE)
    StreamID: 0
    Data:     <zero-filled bytes of specified size>
}
```

受信側はデータを読み取り静かに破棄する必要があります。ウェイストフレームデータは CPU コストを最小化するため意図的にゼロ充填されます（ランダムではない）— TLS レイヤーが既に暗号化を行います。

## コメント

`#` で始まる行はコメントとして扱われ、無視されます：

```
# 高帯域幅接続用カスタムパディング
stop=4
0=50-50
1=200-800
2=500-2000
3=1000-4000
keepalive=60000-120000:4-8
```
