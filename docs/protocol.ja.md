# TSUNAMI プロトコル仕様

> [English](protocol.md) | [中文](protocol.zh.md) | **日本語**

> バージョン 3（現行）

## 概要

TSUNAMI は TLS 1.3 接続内で動作する多重化プロキシプロトコルです。このプロトコルは、単一の暗号化トランスポート上で複数の独立したプロキシストリームを運ぶための軽量バイナリフレーミングレイヤーを定義します。

### 設計目標

| 目標 | アプローチ |
|:-----|:---------|
| **検出不可** | 純粋な TLS 1.3 + ALPN `h2`、ポート 443 — HTTPS と同一 |
| **低オーバーヘッド** | 7 バイトフレームヘッダー、二重暗号化レイヤーなし |
| **アクティブプローブ耐性** | 認証失敗時に実際の HTTP バックエンドに透過的にフォールバック |
| **適応可能なトラフィック形状** | サーバープッシュのパディングスキームでクライアント更新なしにパケットサイズを制御 |
| **高スループット** | BBR 輻輳制御、4 MB バッファ、アダプティブマルチ接続スケーリング |

## 接続ライフサイクル

```
クライアント                                サーバー
  │                                          │
  │──────── TLS 1.3 ハンドシェイク ────────►│  ← ALPN: h2、最低 TLS 1.3
  │◄─────── (ServerHello, Certificate) ────│
  │                                          │
  │──────── 認証: SHA-256(pwd) + パディング ►│  ← 定時間検証
  │                                          │     ┌─ 成功：続行
  │                                          │     └─ 失敗：HTTP バックエンドにフォールバック
  │◄─────── SETTINGS（パディングスキーム）──│
  │◄─────── SERVER_SETTINGS ───────────────│
  │                                          │
  │──────── SYN（ストリーム N）─────────────►│  ← 単調増加の streamId
  │◄─────── SYNACK（ストリーム N）──────────│
  │──────── PSH: ATYP|addr|port ─────────►│  ← 最初の PSH = ターゲットアドレス
  │◄──────► PSH（双方向データ）─────────────►│
  │──────── FIN（ストリーム N）─────────────►│  ← FIN-ACK 不要
  │                                          │
```

## 認証

認証は TLS ハンドシェイク完了後、セッションフレームの前に即座に実行されます。

### 認証リクエスト（クライアント → サーバー）

```
+-------------------+-------------------+-----------+
| SHA-256(password) | padding length    | padding   |
+-------------------+-------------------+-----------+
| 32 bytes          | uint16 big-endian | variable  |
+-------------------+-------------------+-----------+
```

- **SHA-256(password)**：ユーザーパスワードの 32 バイトハッシュ
- **padding length**：後続のランダムパディングの長さ（0–65535）
- **padding**：固定サイズの認証フィンガープリントを防ぐためのランダムバイト

パスワードハッシュは**定時間比較**（`crypto/subtle.ConstantTimeCompare`）で検証され、タイミング攻撃を防止します。

### 認証失敗 — フォールバック

認証失敗時、サーバーはエラーレスポンスを**送信しません**。代わりに、接続全体（認証試行中に既に読み取ったバイトを含む）を設定されたフォールバック HTTP バックエンドに透過的にプロキシします。

これにより：
- アクティブプローバーがサーバーを通常の HTTPS サーバーと区別できない
- プロトコル固有のエラーメッセージが公開されない
- バックエンドのレスポンス（TLS 証明書と HTTP 動作を含む）が実際の Web サイトと一致
- フォールバックバックエンドが設定されていない場合、サーバーは最小限の "It works!" HTML ページで応答し接続をドレイン

## フレームフォーマット

すべてのセッションレイヤー通信は固定サイズの 7 バイトフレームヘッダーを使用します：

```
+----------+-------------------+-------------------+------+
| command  | streamId          | data length       | data |
+----------+-------------------+-------------------+------+
| uint8    | uint32 big-endian | uint16 big-endian | var  |
+----------+-------------------+-------------------+------+
         フレームヘッダーオーバーヘッド合計：7 バイト
```

| フィールド | サイズ | 説明 |
|:------|:-----|:------------|
| `command` | 1 バイト | コマンドタイプ識別子 |
| `streamId` | 4 バイト | ストリーム識別子（ビッグエンディアン） |
| `data length` | 2 バイト | ペイロード長（ビッグエンディアン）、最大 65535 |
| `data` | 可変 | ペイロードデータ（空の場合あり） |

## コマンド

### バージョン 1

| コード | 名前 | 方向 | 説明 |
|:-----|:-----|:---------|:------------|
| `0x00` | `WASTE` | 双方向 | パディングフレーム。受信側はデータを読み取り静かに破棄する必要がある。 |
| `0x01` | `SYN` | クライアント → サーバー | 新しいストリームを開く。`streamId` はセッション内で単調増加する必要がある。 |
| `0x02` | `PSH` | 双方向 | ストリームペイロードデータを運ぶ。 |
| `0x03` | `FIN` | 双方向 | ストリームを閉じる。応答 FIN は不要。 |
| `0x04` | `SETTINGS` | クライアント → サーバー | クライアント設定、新しいセッションごとの最初のフレーム。 |
| `0x05` | `ALERT` | サーバー → クライアント | サーバー警告メッセージ、双方がセッションを閉じる。 |
| `0x06` | `UPDATE_PADDING` | サーバー → クライアント | 新しいパディングスキームをクライアントにプッシュ。 |

### バージョン 2

| コード | 名前 | 方向 | 説明 |
|:-----|:-----|:---------|:------------|
| `0x07` | `SYNACK` | サーバー → クライアント | ストリーム開通を確認、拒否時はデータフィールドにエラーを含む。 |
| `0x08` | `HEART_REQ` | 双方向 | キープアライブ ping。 |
| `0x09` | `HEART_RSP` | 双方向 | キープアライブ pong（`HEART_REQ` へのレスポンス）。 |
| `0x0A` | `SERVER_SETTINGS` | サーバー → クライアント | サーバー機能と設定。 |

### バージョン 3（TSUNAMI 拡張）

| コード | 名前 | 方向 | 説明 |
|:-----|:-----|:---------|:------------|
| `0x0B` | `SURGE_CTRL` | 双方向 | Surge 輻輳制御シグナリング。 |
| `0x0C` | `BW_REPORT` | 双方向 | 定期的な帯域幅統計。 |
| `0x0D` | `STREAM_PRIORITY` | クライアント → サーバー | ストリーム優先度の設定（0=最高、255=最低）。 |

未知のコマンドは前方互換性のため静かに無視する必要があります。

## 設定交換

### クライアント設定（`SETTINGS`、`0x04`）

認証後の最初のフレームとして送信。改行区切りのキーバリューペア形式：

```
v=3
client=http-client/1.0
padding-md5=a1b2c3d4e5f6...
surge-bandwidth=100
```

| キー | タイプ | 説明 |
|:----|:-----|:------------|
| `v` | int | プロトコルバージョン |
| `client` | string | クライアント識別子 |
| `padding-md5` | string | クライアントの現在のパディングスキームの MD5 ハッシュ |
| `surge-bandwidth` | int | クライアント帯域幅（Mbps）（0 = 無効） |

### サーバー設定（`SERVER_SETTINGS`、`0x0A`）

クライアント設定へのレスポンスとして送信：

```
v=3
surge-mode=auto
max-connections=4
threshold=8
```

| キー | タイプ | 説明 |
|:----|:-----|:------------|
| `v` | int | プロトコルバージョン |
| `surge-mode` | string | Surge モード：`none` または `auto` |
| `max-connections` | int | 許可される最大 TLS 接続数（Surge Layer 2） |
| `threshold` | int | Layer 2 アップグレードの同時ストリーム閾値 |

### 設定ネゴシエーションフロー

1. クライアントがプロトコルバージョンと現在のパディングスキームハッシュを含む `SETTINGS` を送信
2. サーバーが Surge パラメータを含む `SERVER_SETTINGS` で応答
3. クライアントの `padding-md5` がサーバーのスキームと異なる場合、サーバーが完全な新スキームを含む `UPDATE_PADDING` を送信
4. クライアントが新しいパディングスキームを即座に適用

## ストリームライフサイクル

1. **オープン**：クライアントが新しい単調増加の `streamId` で `SYN` を送信
2. **確認**：サーバーが `SYNACK` で応答（空データ = 成功、非空 = エラーメッセージ）
3. **ターゲット**：クライアントが SOCKS5 スタイルのターゲットアドレスを含む最初の `PSH` を送信
4. **データ**：双方がプロキシペイロードを運ぶ `PSH` フレームを交換
5. **クローズ**：いずれかの側が `FIN` を送信、確認応答不要

ストリームは `io.ReadWriteCloser` を実装。最大フレームサイズ（65535 バイト）を超えるデータは自動的に複数の `PSH` フレームに分割されます。

## プロキシプロトコル

### TCP プロキシ

新しいストリームの最初の `PSH` フレームには SOCKS5 スタイルのターゲットアドレスが含まれます：

```
+------+----------+-------+
| ATYP | address  | port  |
+------+----------+-------+
| 1B   | variable | u16be |
+------+----------+-------+
```

| ATYP | 値 | アドレスフォーマット |
|:-----|:------|:---------------|
| IPv4 | `0x01` | 4 バイト |
| ドメイン | `0x03` | 1 バイト長 + ドメイン文字列 |
| IPv6 | `0x04` | 16 バイト |

ターゲットアドレスフレームの後、すべての後続 `PSH` フレームは双方向で生の TCP ペイロードを運びます。

### UDP-over-TCP（UoT v2）

UDP トラフィックは UoT v2 フレーミングを使用して TCP ストリーム上で運ばれます。マジックドメイン `sp.v2.udp-over-tcp.arpa` をターゲットとするストリームは UDP リレーモードを示します。

各 UDP パケットはストリーム内で独立してフレーミングされます：

```
+------+----------+-------+--------+------+
| ATYP | address  | port  | length | data |
+------+----------+-------+--------+------+
| 1B   | variable | u16be | u16be  | var  |
+------+----------+-------+--------+------+
```

ATYP/address/port フィールドは各個別パケットの UDP 宛先を指定し、単一ストリーム上での多重化 UDP トラフィックを可能にします。

## トランスポート

### TLS 設定

| パラメータ | 値 | 理由 |
|:----------|:------|:----------|
| 最低バージョン | TLS 1.3 | 前方秘匿性、ダウングレード攻撃なし |
| ALPN | `h2` | HTTP/2 トラフィックパターンを模倣 |
| 暗号スイート | TLS 1.3 デフォルト | AES-GCM / ChaCha20-Poly1305 |
| 証明書 | ユーザー提供または自動自己署名 | 自動生成は ECDSA P-256 |

### TCP チューニング

| パラメータ | デフォルト | 説明 |
|:----------|:--------|:------------|
| `TCP_NODELAY` | 有効 | Nagle アルゴリズムを無効化し低レイテンシーを実現 |
| `SO_SNDBUF` | 4 MB | 高スループットのための大きな送信バッファ |
| `SO_RCVBUF` | 4 MB | 高スループットのための大きな受信バッファ |
| `TCP_KEEPALIVE` | 30s | デッド接続の検出 |
| 輻輳制御 | BBR | Linux のみ、`TCP_CONGESTION` setsockopt で設定 |

> **注意**：BBR はベストエフォートで設定されます。カーネルに BBR がロードされていない場合（`modprobe tcp_bbr`）、システムデフォルトの輻輳制御が静かに使用されます。

## Surge 制御アクション

| コード | 名前 | 説明 |
|:-----|:-----|:------------|
| `0x01` | `ReportThroughput` | 現在のスループットメトリクスを報告 |
| `0x02` | `RequestMoreConn` | 追加の TLS 接続を要求 |
| `0x03` | `ReduceConn` | 接続数の削減を通知 |
| `0x04` | `BandwidthLimit` | 帯域幅制限を設定 |

## セキュリティに関する考慮事項

| 攻撃ベクター | 緩和策 |
|:--------------|:-----------|
| **パスワードブルートフォース** | SHA-256 ハッシュ + 定時間比較、フォールバックによりブルートフォースが通常トラフィックと区別不可能に |
| **アクティブプロービング** | HTTP バックエンドへの透過フォールバック、プロトコル固有のエラーレスポンスなし |
| **トラフィックフィンガープリント** | プログラマブルパディング、ランダム化された認証パディング長 |
| **タイミング分析** | 定時間認証、ランダムキープアライブ間隔 |
| **接続相関** | デフォルト単一接続、オンデマンドでのみマルチ接続 |
| **リプレイ攻撃** | TLS 1.3 がセッションごとの一意な鍵による組み込みリプレイ保護を提供 |
