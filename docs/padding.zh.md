# 可编程填充系统

> [English](padding.md) | **中文** | [日本語](padding.ja.md)

## 概述

TSUNAMI 的可编程填充系统允许服务端动态控制会话前 N 个数据包的填充和分段方式。填充方案通过 `cmdUpdatePaddingScheme`（`0x06`）从服务端推送到所有已连接的客户端，无需任何客户端更新即可更改流量模式。

### 为什么需要可编程填充？

传统代理协议具有固定的数据包大小和时序模式，统计分析可以进行指纹识别。TSUNAMI 采用不同的方法：

- **服务端控制**：服务端定义所有填充规则，而非客户端。调整在一个 RTT 内传播到所有已连接的客户端。
- **逐包粒度**：前 N 个数据包中的每一个都可以有自己的大小分布和分段规则。
- **运行时可更新**：服务端可以随时通过 `UPDATE_PADDING` 命令推送新的填充方案。客户端立即应用。
- **空闲期伪装**：后台保活帧在空闲期间维持流量存在，防止"静默指纹"。

## 方案格式

填充方案定义为换行符分隔的文本块，包含键值对：

```
stop=8
0=30-30
1=100-400
2=400-500,c,500-1000,c,500-1000,c,500-1000,c,500-1000
3=9-9,500-1000
4=500-1000
5=500-1000
6=500-1000
7=500-1000
keepalive=30000-60000:4-8
```

## 键

### `stop`

```
stop=<N>
```

在前 N 个数据包（从 0 开始）后停止应用填充规则。索引 ≥ N 的数据包将不做修改地发送。

**默认值**：`8`

### 数据包索引规则

```
<index>=<segment>[,<segment>,...]
```

定义给定索引的数据包应如何分段。每个数据包索引映射到一个段列表，描述如何拆分和填充外发数据。

### 段类型

| 类型 | 语法 | 描述 |
|:-----|:-------|:------------|
| 大小范围 | `min-max` | 发出大小在 `[min, max]` 字节范围内的随机段 |
| 检查 | `c` | 如果所有用户数据已被消耗，停止发出段 |

**示例**：`2=400-500,c,500-1000,c,500-1000`

对于数据包索引 2：
1. 发出一个 400–500 随机字节的段（填充真实数据 + 填充）
2. **检查**：如果没有剩余用户数据，停止 — 数据包完成
3. 发出一个 500–1000 字节的段
4. **检查**：如果没有剩余用户数据，停止
5. 发出最后一个 500–1000 字节的段

检查（`c`）机制允许短载荷产生较小的数据包，而长载荷可以填充整个分段帧。这创造了抗统计分析的自然变化。

### `keepalive`

```
keepalive=<interval_min>-<interval_max>:<size_min>-<size_max>
```

配置空闲期后台保活数据包。当会话中的所有流都空闲时，客户端以随机间隔发送小的 `cmdWaste` 填充帧。

| 参数 | 描述 |
|:----------|:------------|
| `interval_min`–`interval_max` | 保活数据包之间的随机间隔（毫秒） |
| `size_min`–`size_max` | 随机保活数据包大小（字节） |

**示例**：`keepalive=30000-60000:4-8` 在空闲时每 30–60 秒发送一个 4–8 字节的废弃帧。

## 默认方案

内置默认方案：

```
stop=8
0=30-30
1=100-400
2=400-500,c,500-1000,c,500-1000,c,500-1000,c,500-1000
3=9-9,500-1000
4=500-1000
5=500-1000
6=500-1000
7=500-1000
keepalive=30000-60000:4-8
```

此方案：
- 填充每个会话的前 8 个数据包
- 确保第一个数据包正好是 30 字节（与认证填充大小匹配）
- 对数据包 2–7 应用可变长度分段和检查点
- 在空闲期间每 30–60 秒发送微小的保活帧

### 调优指南

| 场景 | 调整 |
|:---------|:-----------|
| 高带宽服务器 | 增加 `stop`（如 12–16）；使用更大的大小范围 |
| 低带宽/移动端 | 减少 `stop`（如 4）；使用更小的大小范围以减少开销 |
| 高安全环境 | 添加更多带 `c` 检查点的段；使用更短的保活间隔 |
| CDN 前置部署 | 使填充大小匹配典型的 HTTP/2 响应模式 |

## 方案版本控制

每个方案由其原始文本的 **MD5 哈希**标识。客户端在 `SETTINGS` 帧中包含其当前的 `padding-md5`。如果服务端的方案不同，它会通过 `cmdUpdatePaddingScheme` 推送新方案。

这避免了在重连时不必要地重传未变化的方案。

## 线路协议

### 推送新方案

服务端发送 `cmdUpdatePaddingScheme`（`0x06`），帧数据为原始方案文本：

```
Frame {
    Command:  0x06 (UPDATE_PADDING)
    StreamID: 0
    Data:     <scheme text bytes>
}
```

客户端解析方案、计算其 MD5，并立即将其应用于所有后续数据包。

### 填充帧

填充使用 `cmdWaste`（`0x00`）帧实现：

```
Frame {
    Command:  0x00 (WASTE)
    StreamID: 0
    Data:     <zero-filled bytes of specified size>
}
```

接收方必须读取数据并静默丢弃。废弃帧数据有意使用零填充（而非随机）以最小化 CPU 开销 — TLS 层已经对其进行了加密。

## 注释

以 `#` 开头的行被视为注释并被忽略：

```
# 高带宽连接的自定义填充
stop=4
0=50-50
1=200-800
2=500-2000
3=1000-4000
keepalive=60000-120000:4-8
```
