# TSUNAMI 协议规范

> [English](protocol.md) | **中文** | [日本語](protocol.ja.md)

> 版本 3（当前）

## 概述

TSUNAMI 是一个在 TLS 1.3 连接内运行的多路复用代理协议。该协议定义了一个轻量级的二进制帧层，用于在单个加密传输上承载多个独立的代理流。

### 设计目标

| 目标 | 方法 |
|:-----|:---------|
| **不可检测** | 纯 TLS 1.3 + ALPN `h2`，端口 443 — 与 HTTPS 线路完全一致 |
| **低开销** | 7 字节帧头；无重复加密层 |
| **抗主动探测** | 认证失败时透明回退到真实 HTTP 后端 |
| **可调整流量特征** | 服务端推送的填充方案，无需客户端更新即可控制数据包大小 |
| **高吞吐量** | BBR 拥塞控制、4 MB 缓冲区、自适应多连接扩展 |

## 连接生命周期

```
客户端                                     服务端
  │                                          │
  │──────── TLS 1.3 握手 ──────────────────►│  ← ALPN: h2，最低 TLS 1.3
  │◄─────── (ServerHello, Certificate) ────│
  │                                          │
  │──────── 认证: SHA-256(pwd) + 填充 ────►│  ← 恒定时间验证
  │                                          │     ┌─ 成功：继续
  │                                          │     └─ 失败：回退到 HTTP 后端
  │◄─────── SETTINGS（填充方案）───────────│
  │◄─────── SERVER_SETTINGS ───────────────│
  │                                          │
  │──────── SYN（流 N）──────────────────►│  ← 单调递增的 streamId
  │◄─────── SYNACK（流 N）────────────────│
  │──────── PSH: ATYP|addr|port ─────────►│  ← 第一个 PSH = 目标地址
  │◄──────► PSH（双向数据）───────────────►│
  │──────── FIN（流 N）──────────────────►│  ← 无需 FIN-ACK
  │                                          │
```

## 认证

认证在 TLS 握手完成后、任何会话帧之前立即执行。

### 认证请求（客户端 → 服务端）

```
+-------------------+-------------------+-----------+
| SHA-256(password) | padding length    | padding   |
+-------------------+-------------------+-----------+
| 32 bytes          | uint16 big-endian | variable  |
+-------------------+-------------------+-----------+
```

- **SHA-256(password)**：用户密码的 32 字节哈希
- **padding length**：后续随机填充的长度（0–65535）
- **padding**：随机字节，防止固定大小的认证指纹

密码哈希使用**恒定时间比较**（`crypto/subtle.ConstantTimeCompare`）进行验证，以防止时序攻击。

### 认证失败 — 回退

认证失败时，服务端**不会**发送任何错误响应。相反，它会将整个连接（包括在认证尝试期间已读取的字节）透明代理到配置的回退 HTTP 后端。

这确保了：
- 主动探测者无法将服务器与正常 HTTPS 服务器区分
- 不会暴露任何协议特定的错误信息
- 后端的响应（包括 TLS 证书和 HTTP 行为）与真实网站匹配
- 如果未配置回退后端，服务端会响应一个最小的"It works!"HTML 页面并排空连接

## 帧格式

所有会话层通信使用固定大小的 7 字节帧头：

```
+----------+-------------------+-------------------+------+
| command  | streamId          | data length       | data |
+----------+-------------------+-------------------+------+
| uint8    | uint32 big-endian | uint16 big-endian | var  |
+----------+-------------------+-------------------+------+
         总帧头开销：7 字节
```

| 字段 | 大小 | 描述 |
|:------|:-----|:------------|
| `command` | 1 字节 | 命令类型标识符 |
| `streamId` | 4 字节 | 流标识符（大端序） |
| `data length` | 2 字节 | 载荷长度（大端序），最大 65535 |
| `data` | 可变 | 载荷数据（可为空） |

## 命令

### 版本 1

| 代码 | 名称 | 方向 | 描述 |
|:-----|:-----|:---------|:------------|
| `0x00` | `WASTE` | 双向 | 填充帧。接收方必须读取并静默丢弃数据。 |
| `0x01` | `SYN` | 客户端 → 服务端 | 打开新流。`streamId` 在会话内必须单调递增。 |
| `0x02` | `PSH` | 双向 | 承载流载荷数据。 |
| `0x03` | `FIN` | 双向 | 关闭流。无需回复 FIN。 |
| `0x04` | `SETTINGS` | 客户端 → 服务端 | 客户端设置，每个新会话的第一个帧。 |
| `0x05` | `ALERT` | 服务端 → 客户端 | 服务端警告信息；双方关闭会话。 |
| `0x06` | `UPDATE_PADDING` | 服务端 → 客户端 | 向客户端推送新的填充方案。 |

### 版本 2

| 代码 | 名称 | 方向 | 描述 |
|:-----|:-----|:---------|:------------|
| `0x07` | `SYNACK` | 服务端 → 客户端 | 确认流已打开；数据字段在拒绝时携带错误信息。 |
| `0x08` | `HEART_REQ` | 双向 | 保活 ping。 |
| `0x09` | `HEART_RSP` | 双向 | 保活 pong（对 `HEART_REQ` 的响应）。 |
| `0x0A` | `SERVER_SETTINGS` | 服务端 → 客户端 | 服务端能力和配置。 |

### 版本 3（TSUNAMI 扩展）

| 代码 | 名称 | 方向 | 描述 |
|:-----|:-----|:---------|:------------|
| `0x0B` | `SURGE_CTRL` | 双向 | Surge 拥塞控制信令。 |
| `0x0C` | `BW_REPORT` | 双向 | 周期性带宽统计。 |
| `0x0D` | `STREAM_PRIORITY` | 客户端 → 服务端 | 设置流优先级（0=最高，255=最低）。 |

未知命令必须静默忽略以保持前向兼容性。

## 设置交换

### 客户端设置（`SETTINGS`，`0x04`）

认证后作为第一帧发送。格式为换行符分隔的键值对：

```
v=3
client=tsunami-client/1.0
padding-md5=a1b2c3d4e5f6...
surge-bandwidth=100
```

| 键 | 类型 | 描述 |
|:----|:-----|:------------|
| `v` | int | 协议版本 |
| `client` | string | 客户端标识符 |
| `padding-md5` | string | 客户端当前填充方案的 MD5 哈希 |
| `surge-bandwidth` | int | 客户端带宽（Mbps）（0 = 禁用） |

### 服务端设置（`SERVER_SETTINGS`，`0x0A`）

作为对客户端设置的响应发送：

```
v=3
surge-mode=auto
max-connections=4
threshold=8
```

| 键 | 类型 | 描述 |
|:----|:-----|:------------|
| `v` | int | 协议版本 |
| `surge-mode` | string | Surge 模式：`none` 或 `auto` |
| `max-connections` | int | 允许的最大 TLS 连接数（Surge Layer 2） |
| `threshold` | int | Layer 2 升级的并发流阈值 |

### 设置协商流程

1. 客户端发送 `SETTINGS`，包含协议版本和当前填充方案哈希
2. 服务端响应 `SERVER_SETTINGS`，包含 Surge 参数
3. 如果客户端的 `padding-md5` 与服务端方案不同，服务端发送 `UPDATE_PADDING` 携带完整的新方案
4. 客户端立即应用新的填充方案

## 流生命周期

1. **打开**：客户端发送 `SYN`，带有新的单调递增 `streamId`
2. **确认**：服务端响应 `SYNACK`（空数据 = 成功，非空 = 错误信息）
3. **目标**：客户端发送第一个 `PSH`，包含 SOCKS5 风格的目标地址
4. **数据**：双方交换携带代理载荷的 `PSH` 帧
5. **关闭**：任一方发送 `FIN`；无需确认

流实现 `io.ReadWriteCloser`。超过最大帧大小（65535 字节）的数据会自动拆分为多个 `PSH` 帧。

## 代理协议

### TCP 代理

新流上的第一个 `PSH` 帧包含 SOCKS5 风格的目标地址：

```
+------+----------+-------+
| ATYP | address  | port  |
+------+----------+-------+
| 1B   | variable | u16be |
+------+----------+-------+
```

| ATYP | 值 | 地址格式 |
|:-----|:------|:---------------|
| IPv4 | `0x01` | 4 字节 |
| 域名 | `0x03` | 1 字节长度 + 域名字符串 |
| IPv6 | `0x04` | 16 字节 |

目标地址帧之后，所有后续 `PSH` 帧双向承载原始 TCP 载荷。

### UDP-over-TCP（UoT v2）

UDP 流量通过 UoT v2 帧在 TCP 流上承载。目标为魔术域名 `sp.v2.udp-over-tcp.arpa` 的流表示 UDP 中继模式。

每个 UDP 数据包在流中独立帧化：

```
+------+----------+-------+--------+------+
| ATYP | address  | port  | length | data |
+------+----------+-------+--------+------+
| 1B   | variable | u16be | u16be  | var  |
+------+----------+-------+--------+------+
```

ATYP/address/port 字段指定每个单独数据包的 UDP 目标，允许在单个流上多路复用 UDP 流量。

## 传输

### TLS 配置

| 参数 | 值 | 原因 |
|:----------|:------|:----------|
| 最低版本 | TLS 1.3 | 前向安全，无降级攻击 |
| ALPN | `h2` | 模拟 HTTP/2 流量模式 |
| 密码套件 | TLS 1.3 默认值 | AES-GCM / ChaCha20-Poly1305 |
| 证书 | 用户提供或自动自签名 | 自动生成使用 ECDSA P-256 |

### TCP 调优

| 参数 | 默认值 | 描述 |
|:----------|:--------|:------------|
| `TCP_NODELAY` | 启用 | 禁用 Nagle 算法以降低延迟 |
| `SO_SNDBUF` | 4 MB | 大发送缓冲区以提高吞吐量 |
| `SO_RCVBUF` | 4 MB | 大接收缓冲区以提高吞吐量 |
| `TCP_KEEPALIVE` | 30s | 检测死连接 |
| 拥塞控制 | BBR | 仅 Linux；通过 `TCP_CONGESTION` setsockopt 设置 |

> **注意**：BBR 为尽力设置。如果内核未加载 BBR（`modprobe tcp_bbr`），则静默使用系统默认拥塞控制。

## Surge 控制动作

| 代码 | 名称 | 描述 |
|:-----|:-----|:------------|
| `0x01` | `ReportThroughput` | 报告当前吞吐量指标 |
| `0x02` | `RequestMoreConn` | 请求额外的 TLS 连接 |
| `0x03` | `ReduceConn` | 信号减少连接数 |
| `0x04` | `BandwidthLimit` | 设置带宽上限 |

## 安全考虑

| 攻击向量 | 缓解措施 |
|:--------------|:-----------|
| **密码暴力破解** | SHA-256 哈希 + 恒定时间比较；回退使暴力破解与正常流量无法区分 |
| **主动探测** | 透明回退到 HTTP 后端；无协议特定错误响应 |
| **流量指纹** | 可编程填充；随机化认证填充长度 |
| **时序分析** | 恒定时间认证；随机保活间隔 |
| **连接关联** | 默认单连接；仅按需多连接 |
| **重放攻击** | TLS 1.3 通过唯一的每会话密钥提供内置重放保护 |
