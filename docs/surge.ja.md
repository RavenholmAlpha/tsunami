# Surge：レイヤード輻輳制御

> [English](surge.md) | [中文](surge.zh.md) | **日本語**

## 概要

Surge は TSUNAMI のアダプティブ接続スケーリングシステムです。リアルタイムのストリーム同時実行数に基づいて、基盤となる TLS 接続数を管理し、単一接続の効率とマルチ接続のスループットのバランスを取ります。

### なぜ Surge か？

単一接続の多重化はステルス性に理想的です（1 接続 = 1 HTTPS セッション）。しかし高負荷（多数の同時ストリーム）では、単一の TLS 接続がボトルネックになります：

- **ヘッドオブラインブロッキング**：1 つのストリームでの遅い TLS 書き込みが他のすべてをブロック
- **TCP フェアネス**：1 つの接続は帯域幅の 1 つの「公平なシェア」しか得られない。複数接続で帯域幅を集約可能
- **キープアライブオーバーヘッド**：1 つの接続上の多数のアイドルストリームが不要なフレーム処理を発生

Surge は実際の需要に基づいて単一接続（Layer 1）とマルチ接続（Layer 2）を動的に切り替えることでこれを解決します。

## 設計原則

1. **デフォルトは単一接続** — 低負荷時のリソース使用とフィンガープリントを最小化
2. **自動スケーリング** — 必要な時のみ複数接続にアップグレード
3. **パケット並べ替えなし** — 各ストリームはライフタイム全体で 1 つの TLS 接続に固定
4. **単一ポート** — すべての接続が同じポートを使用（通常 443）
5. **段階的ティアダウン** — アイドル接続は設定可能なアイドルタイムアウト後にのみ閉じる

## レイヤードアーキテクチャ

### Layer 1（デフォルト）

すべてのストリームが単一の TLS 接続上で多重化されます。

```
Stream A ─┐
Stream B ─┤──► Session 0 ──► TLS Connection 0 ──► :443
Stream C ─┘
```

**アクティブ条件**：同時ストリーム数 ≤ 閾値（デフォルト：8）

### Layer 2（自動アップグレード）

同時ストリーム数が閾値を超えると、新しい TLS 接続が開かれ、最小負荷戦略でストリームが分散されます。

```
Stream A ─┐
Stream B ─┤──► Session 0 ──► TLS Connection 0 ──┐
Stream C ─┘                                      │
                                                  ├──► :443
Stream D ─┐                                      │
Stream E ─┤──► Session 1 ──► TLS Connection 1 ──┘
Stream F ─┘
```

**アクティブ条件**：同時ストリーム数 > 閾値

### レイヤー遷移

```
         ストリーム数 > 閾値
Layer 1 ─────────────────────────► Layer 2
         ◄─────────────────────────
         ストリーム数 ≤ 閾値/4
         かつセッション数が 1 に減少
```

- **アップグレード**（1 → 2）：アクティブストリーム数が閾値を超えるとトリガー。コントローラーは `max-connections` まで追加セッションを非同期で事前作成。
- **ダウングレード**（2 → 1）：アクティブストリームが `threshold/4` 以下に減少するとトリガー（急速な振動を防ぐためのワイドヒステリシス）。アイドルセッション（`idle-downgrade-time` より長くアイドル）が閉じられ、少なくとも 1 つは保持。セッション数が 1 になって初めて Layer 1 に復帰。

コントローラーは **2 秒**ごとにレイヤー遷移を評価します。

### アップグレード動作

Layer 2 にアップグレードする際、Surge は 1 つの追加接続だけでなく、`max-connections` まで**事前拡張**します：

```
evaluate() がストリーム数 (13) > 閾値 (8) を検出
  → Layer 2 を保存
  → 非同期：プールが max-connections (4) になるまでセッションを作成
    → Session 2 作成 (2/4)
    → Session 3 作成 (3/4)
    → Session 4 作成 (4/4)
```

Layer 2 中にストリーム圧力が続く場合、コントローラーは `max-connections` に達するまで各評価サイクルで拡張を試行し続けます。

## 設定

| パラメータ | デフォルト | 説明 |
|:----------|:--------|:------------|
| `mode` | `auto` | 動作モード：`none`（単一接続のみ）または `auto` |
| `threshold` | `8` | Layer 2 をトリガーする同時ストリーム数 |
| `max-connections` | `4` | Layer 2 での最大 TLS 接続数 |
| `idle-downgrade-time` | `30s` | ダウングレード時にクローズ対象となるまでのセッションアイドル時間 |

### クライアント CLI

```bash
./tsunami-client \
  --server example.com:443 \
  --password "..." \
  --max-connections 4 \
  --threshold 8
```

### チューニングガイドライン

| シナリオ | 推奨 |
|:---------|:---------------|
| 軽量ブラウジング（< 8 タブ） | デフォルト設定（threshold=8, max=4）で十分 |
| 大量の同時ダウンロード | 閾値を 16 に増加、max-connections を 8 に増加 |
| モバイル/従量課金接続 | モードを `none` に設定して単一接続を維持 |
| 高レイテンシーリンク | 閾値を 4 に下げてより早くアップグレード |

## セッション選択

### Layer 1

標準のセッションプールロジックを使用：アイドルセッションを優先（最新優先）、多重化のために非クローズセッションにフォールバック。セッションが存在しない場合のみ新規作成。

### Layer 2

**最小負荷**分散を使用：新しいストリームはアクティブストリームが最も少ないセッションに割り当て。利用可能なセッションがない場合、新規作成（最大 `max-connections`）。

これにより、アクティブなマイグレーションなしに接続間で均等なストリーム分散が保証されます（ストリームはセッション間で移動しません）。

## セッションプール

セッションプール（`pkg/mux`）はすべての TLS 接続のライフサイクルを管理します：

| 動作 | 詳細 |
|:---------|:-------|
| **アイドルチェック** | 30 秒ごとに、60 秒以上アイドルのセッションを閉じる（少なくとも 1 つ保持） |
| **シーケンス追跡** | 各セッションに順序付け用の単調増加シーケンス番号 |
| **自動作成** | 適切なセッションがない場合、プールが新しい TLS 接続をダイヤル |
| **グレースフルクローズ** | アクティブストリームは自然に完了、アイドルセッションのみクリーンアップ対象 |

## ワイヤプロトコル

Surge は制御シグナリングに `SURGE_CTRL`（`0x0B`）フレームを使用します：

| アクションコード | 名前 | 説明 |
|:------------|:-----|:------------|
| `0x01` | `ReportThroughput` | 現在のスループットメトリクスを報告 |
| `0x02` | `RequestMoreConn` | クライアントに追加接続のオープンを要求 |
| `0x03` | `ReduceConn` | クライアントに接続数の削減を通知 |
| `0x04` | `BandwidthLimit` | 帯域幅制限を設定 |

帯域幅統計は `BW_REPORT`（`0x0C`）フレームを通じて定期的に交換されます。

## 代替手段との比較

| アプローチ | 長所 | 短所 |
|:---------|:-----|:-----|
| **常に単一接続** | 最小フィンガープリント、シンプル | ヘッドオブラインブロッキング、帯域幅ボトルネック |
| **常にマルチ接続** | 高スループット | 大きなフィンガープリント、低負荷時のリソース浪費 |
| **Surge（アダプティブ）** | 両方の利点：低負荷時はステルス、高負荷時はスループット | レイヤー遷移ロジックのわずかな複雑さ |
